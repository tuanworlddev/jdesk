# JDesk

JDesk is a desktop application framework built with a Java 25 application core, a web
frontend, and the native WebView supplied by each operating system.

Applications keep business logic, system integration, lifecycle handling, and packaging
in Java. The user interface can use vanilla HTML/TypeScript or frameworks such as React,
Vue, and Svelte.

## Features

- Native system WebViews: WebView2 on Windows, WKWebView on macOS, and WebKitGTK on Linux.
- Java 25, JPMS modules, virtual threads, and Foreign Function & Memory API adapters.
- Asynchronous Java-to-JavaScript commands and events with compile-time generated bindings.
- TypeScript client with request correlation, cancellation, timeouts, and navigation reset.
- Deny-by-default capabilities and per-window permission grants.
- Strict navigation policy and a production-only `jdesk://app/` asset protocol with
  HTTP Range (206), configurable CSP, and app-defined Java-served asset routes.
- Binary pull-streaming with backpressure (`BinaryStream` → `invokeStream`), verified
  live at 2 GiB.
- OS-backed secret storage (macOS Keychain, Windows DPAPI, Linux Secret Service).
- Structured command errors (`error.data`) and a page-console-to-Java logging bridge.
- Opt-in automation endpoint for E2E tests, CI, and agents (`-Djdesk.automation=true`).
- Window minimum size, start-maximized, and remembered bounds across runs.
- Frontend HMR through the configured frontend development server — or a zero-Node dev
  loop with automatic in-app reload for static frontends.
- Automatic Java/resource rebuild and controlled process restart through `jdeskDev`.
- `basic` and `structured` project templates generated by the `jdesk` CLI.
- Runtime images with `jlink` and native application/installer packaging with `jpackage`.
- SHA-256 checksums and a CycloneDX SBOM for packaged applications.

## Requirements

- JDK 25. The JDK must include `jlink` and `jpackage` when building desktop packages.
- Node.js and npm when the selected frontend uses Vite or another Node-based toolchain.
- The platform WebView runtime:
  - Windows: Microsoft Edge WebView2 Runtime.
  - macOS: system WKWebView; no additional runtime installation is required.
  - Linux: WebKitGTK 4.1, for example `sudo apt-get install libwebkit2gtk-4.1-0`.

## Quick Start

Scaffold a new app with the published npm CLI — the same flow as `create-vite` /
`create-next-app`:

```bash
npm create jdesk-app@latest my-app
# or
npx create-jdesk-app@latest my-app
```

It walks you through a guided setup (project name, package id, build system, template) and
prints the commands to run. See [`create-jdesk-app` on npm](https://www.npmjs.com/package/create-jdesk-app).

> **Building a scaffolded app (pre-alpha).** The `dev.jdesk:*` framework artifacts are not on
> Maven Central yet — they are published to GitHub Packages (which requires authentication to
> consume). So to build the generated project today, either scaffold with
> `--jdesk-source /path/to/JDesk` (a local checkout, used as a Gradle composite build), run
> `./gradlew publishToMavenLocal` from a checkout first, or configure the GitHub Packages
> repository with a `read:packages` token. Anonymous `npx create-jdesk-app && ./gradlew run`
> becomes fully turnkey once the artifacts reach Maven Central. See
> [scaffolding and publishing](docs/development/scaffolding-and-publishing.md).

## Install From Source

Alternatively, build the CLI from a local checkout of this repository:

Build and test the framework:

```bash
./gradlew clean check
```

Build the CLI distribution:

```bash
./gradlew :modules:jdesk-cli:installDist
```

The CLI is created at:

```text
modules/jdesk-cli/build/install/jdesk/bin/jdesk
```

On macOS or Linux, add it to the current shell's `PATH`:

```bash
export JDESK_HOME="$PWD"
export PATH="$JDESK_HOME/modules/jdesk-cli/build/install/jdesk/bin:$PATH"
jdesk --help
```

On Windows PowerShell:

```powershell
$env:JDESK_HOME = (Get-Location).Path
$env:Path = "$env:JDESK_HOME\modules\jdesk-cli\build\install\jdesk\bin;$env:Path"
jdesk.bat --help
```

## Create An Application

Use the npm CLI (recommended). Run it with no arguments for a guided setup:

```bash
npx create-jdesk-app@latest
```

It asks for the project name, Java package (default `com.example.<name>`, press Enter to
accept), build system (**Gradle** — full tooling, or **Maven** — build + run), and, for
Gradle, a frontend template (arrow keys): `basic`, `vanilla`, `react`, `vue`, `svelte`, or
`structured`. Or pass everything as flags:

```bash
npx create-jdesk-app@latest my-app --template react --package com.example.myapp
npx create-jdesk-app@latest my-suite --template structured
npx create-jdesk-app@latest my-app --maven          # Maven instead of Gradle
```

Then:

```bash
cd my-app
./gradlew run          # launch the app on this OS (Gradle)
./gradlew dev          # dev loop with frontend hot-reload
```

You can also use the bundled Java CLI from a source checkout
(`jdesk create my-app --template basic --package com.example.myapp --jdesk-source "$JDESK_HOME"`);
`--jdesk-source` wires the generated project to the local framework as a Gradle composite
build (see the pre-alpha note under Quick Start).

Each generated project includes:

- Gradle wrapper + JDK 25 toolchain (or a `pom.xml` for the Maven build system).
- A short `run` task and aliases (`run`, `dev`, `doctor`, `bindings`, `pkg`).
- A capability policy and a sample typed Java command.
- A frontend (plain HTML/JS, or Vite + React/Vue/Svelte) and a production asset builder.
- Development, packaging, and installer tasks from `dev.jdesk.application`.

Single-module templates are classpath apps (no `module-info.java`); the `structured`
template is modular.

## Development

Install the generated frontend dependencies once:

```bash
cd my-app
npm install --prefix ui
```

Start the frontend server and desktop application:

```bash
./gradlew jdeskDev
```

For a structured project:

```bash
./gradlew :desktop:jdeskDev
```

Frontend changes are applied by Vite HMR. Java and resource changes run the project's
`classes` task and restart the Java application only after a successful build. If a build
fails, the existing application remains running while the source is corrected.

Development settings can be customized in `build.gradle.kts`:

```kotlin
jdesk {
    development {
        javaReload.set(true)
        reloadDebounceMillis.set(300)
        // reloadCommand.set(listOf("./gradlew", "classes"))
        // reloadSources.from(rootProject.file("another-module/src/main"))
    }
}
```

## Build And Package

Check the local environment before packaging:

```bash
./gradlew jdeskDoctor
```

Compile the application and frontend:

```bash
./gradlew classes
```

Create a platform application image:

```bash
./gradlew jdeskPackage
```

Create a native installer on the current operating system:

```bash
./gradlew jdeskInstaller
```

The installer task supports DMG/PKG on macOS, MSI/EXE on Windows, and DEB/RPM on Linux.
Current generated installers are unsigned unless signing credentials are configured.
See [packaging and signing](docs/packaging/packaging-and-signing.md).

## Repository Layout

- `modules/jdesk-api`: public Java application API.
- `modules/jdesk-runtime`: lifecycle, IPC, capabilities, assets, and runtime coordination.
- `modules/jdesk-platform-*`: Windows, macOS, and Linux WebView adapters.
- `modules/jdesk-codegen`: Java command registry and TypeScript binding generator.
- `modules/jdesk-gradle-plugin`: development, build, and packaging tasks.
- `modules/jdesk-cli`: project generator and bundled templates.
- `js/jdesk-client`: TypeScript IPC client.
- `examples`: example applications.
- `test-apps`: native, security, and packaging probes.
- `docs`: architecture, development, security, packaging, and verification documentation.

## Project Status

JDesk is pre-alpha and under active development. The core runtime and primary platform
adapters are implemented and tested. The `create-jdesk-app` CLI is published to npm, and the
`dev.jdesk:*` artifacts are published to GitHub Packages (v0.1.1). Maven Central publishing
(for anonymous consumption), signed/notarized releases, secondary CPU architectures, and a
macOS CI job are still pending.

See [IMPLEMENTATION_STATUS.md](IMPLEMENTATION_STATUS.md) and the
[implementation report](docs/verification/final-report.md) for the detailed status.
