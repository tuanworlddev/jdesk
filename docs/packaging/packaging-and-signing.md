# Packaging and Signing

JDesk distributes as a **`jlink` runtime image** wrapped by **`jpackage`** into a
platform-native application image / installer ([spec section 16](../../JDESK_CORE_FRAMEWORK_SPEC.md),
[ADR-007](../architecture/ADR-007-jvm-distribution-first.md)). The runtime image starts
without a globally installed JRE. The argument builders live in `jdesk-packager`; the
Gradle tasks live in the [plugin](../development/gradle-plugin-reference.md).

> **Status.** App-image packaging (`jdeskPackage`) works and is verified on all three
> primary targets. **Installer creation (`jdeskInstaller`) builds a real DMG/MSI/DEB via jpackage
> on the target OS (UNSIGNED without a signing identity)**; signing is a configuration surface, not yet an end-to-end signed pipeline.
> Checksums and the CycloneDX SBOM are generated by `jdeskPackage`. Each item below says
> plainly what is real today.

## Pipeline

1. **Frontend build** — `jdeskFrontendBuild` runs your `buildCommand`; the built
   `distDirectory` is packed into the app jar under `/web`.
2. **Runtime image** — `jdeskRuntimeImage` runs `jdeps --print-module-deps
   --ignore-missing-deps --multi-release <v>` over the runtime classpath, then `jlink`
   into `build/jdesk/runtime-image`. It carries no global native-access privilege.
3. **App image** — `jdeskPackage` stages named modules and runs
   `jpackage --module-path ... --module <mainModule>/<mainClass>`. The launcher grants
   native access only to `dev.jdesk.platform.<os>` and denies all other illegal native
   access. Output is written to
   `build/jdesk/package`. macOS adds `--mac-package-identifier <applicationId>` and
   `-XstartOnFirstThread`.
4. **Package smoke** — `jdeskNativeSmokeTest` launches the packaged app-image's real
   launcher with `--jdesk-smoke` and requires exit 0 (see
   [../verification/native-testing-and-evidence.md](../verification/native-testing-and-evidence.md)).
5. **Installer** — `jdeskInstaller` runs `jpackage` on the app image to build the OS's
   native installer (DMG/PKG on macOS, MSI/EXE on Windows, DEB/RPM on Linux) into
   `build/jdesk/installer`. Type defaults to the OS default; override with
   `-PjdeskInstallerType=<dmg|pkg|msi|exe|deb|rpm>`. UNSIGNED unless a signing identity is
   configured. Verified end-to-end locally (real 34 MB DMG) and in the CI package jobs.

## Build on the target OS only

Cross-packaging is **forbidden** ([spec 16.2](../../JDESK_CORE_FRAMEWORK_SPEC.md)): build
Windows packages on Windows, macOS on macOS, Linux on Linux. `jpackage` produces
platform-native images and cannot be trusted to cross-produce another OS's package. CI has
per-OS package jobs (`package-windows-x64`, `package-linux-x64`; macOS verified locally).

Per-OS target formats (spec 16.2):

- **Windows:** application image + EXE or MSI.
- **macOS:** `.app` + DMG or PKG.
- **Linux:** application image + DEB (RPM may follow).

Both the **application image** and the **installer** steps are implemented: `jdeskPackage`
builds the image, `jdeskInstaller` builds the native installer on the same OS. macOS DMG is
verified locally; Windows MSI and Linux DEB are built in the CI package jobs.

## Signing (configuration hooks)

Signing is applied by the **OS toolchains**, wired through the plugin's signing hooks. The
spec ([16.3](../../JDESK_CORE_FRAMEWORK_SPEC.md)) requires a configuration surface and
documentation for:

- **Windows — Authenticode.** Sign the EXE/MSI with `signtool` and your code-signing
  certificate.
- **macOS — Developer ID + notarization.** `codesign` with a Developer ID Application
  certificate, then `notarytool` submit + `stapler`.
- **Linux — package/repository signing.** e.g. `dpkg-sig`/`debsign` for DEB, repo GPG
  signing.

**CI packages are `UNSIGNED`.** Unsigned packages are allowed for development verification
but are labeled `UNSIGNED` and **do not satisfy a signed-release gate**. A real signed
release requires the certificates/credentials above, which are not present in CI.

> **Honest status.** The signing hooks are a *configuration surface*; a fully wired,
> end-to-end signed-and-notarized release pipeline is **Phase-7 work** and not yet
> demonstrated. Do not treat any current artifact as signed.

## SBOM and checksums

Release-artifact hygiene lives in `jdesk-packager`'s `ReleaseArtifacts`
([spec 12.5, 16](../../JDESK_CORE_FRAMEWORK_SPEC.md)):

- **`writeChecksums(root, checksumFile)`** — SHA-256 for every file under `root`
  (recursively), emitted in GNU coreutils format, sorted for determinism.
- **`writeSbom(sbomFile, applicationId, version, artifacts)`** — a minimal **CycloneDX
  1.5** JSON SBOM listing the application and each checksummed artifact as a component.
  Deterministic (stable name-based serial number, no timestamps); unit-tested for validity
  and byte-stability.

`jdeskPackage` invokes both after building the application image: it writes
`checksums.sha256` (over the produced image) and `sbom.cyclonedx.json` next to the image
and logs `wrote checksums.sha256 (N files) and sbom.cyclonedx.json (UNSIGNED)`. Verified
against a real 282-file jpackage app image. The redistributed `WebView2Loader.dll` license
is documented with Windows artifacts (see
[../platform/prerequisites.md](../platform/prerequisites.md)).

**Still planned.** A dedicated signed-release gate (blocking release when signing hooks are
unset) and an aggregate `jdeskRelease` task are follow-ups; today installers build UNSIGNED
and a signed+notarized end-to-end pipeline is not yet demonstrated.
