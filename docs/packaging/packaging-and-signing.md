# Packaging and Signing

JDesk distributes as a **`jlink` runtime image** wrapped by **`jpackage`** into a
platform-native application image / installer ([spec section 16](../../JDESK_CORE_FRAMEWORK_SPEC.md),
[ADR-007](../architecture/ADR-007-jvm-distribution-first.md)). The runtime image starts
without a globally installed JRE. The argument builders live in `jdesk-packager`; the
Gradle tasks live in the [plugin](../development/gradle-plugin-reference.md).

> **Status.** App-image packaging (`jdeskPackage`) works and is verified on all three
> primary targets. **Installer creation (`jdeskInstaller`) builds a real DMG/MSI/DEB via jpackage
> on the target OS (UNSIGNED without a signing identity)**. Platform signing, verification and
> macOS notarization are wired; a publicly trusted signed run still needs publisher credentials.
> Checksums and the CycloneDX 1.7 SBOM are generated by `jdeskPackage`. Each item below says
> plainly what is real today.

## Pipeline

1. **Frontend build** — `jdeskFrontendBuild` runs your `buildCommand`; the built
   `distDirectory` is packed into the app jar under `/web`.
2. **Runtime image** — `jdeskRuntimeImage` runs `jdeps --print-module-deps
   --ignore-missing-deps --multi-release <v>` over the runtime classpath, then `jlink`
   into `build/jdesk/runtime-image`. It carries no global native-access privilege. jlink runs
   with `--strip-debug` and `--compress=zip-6` by default, which roughly a third off the image
   (measured ~46 MB → ~29 MB) — the bundled runtime is the dominant size lever for a JDesk app.
3. **App image** — `jdeskPackage` stages named modules and runs
   `jpackage --name <applicationName> --module-path ... --module <mainModule>/<mainClass>`.
   The launcher grants native access only to `dev.jdesk.platform.<os>` and denies all other
   illegal native access. Output is written to
   `build/jdesk/package`. macOS adds `--mac-package-identifier <applicationId>` and
   `-XstartOnFirstThread`. See [Application name and the macOS menu bar](#application-name-and-the-macos-menu-bar).
4. **Package smoke** — `jdeskNativeSmokeTest` launches the packaged app-image's real
   launcher with `--jdesk-smoke` and requires exit 0 (see
   [../verification/native-testing-and-evidence.md](../verification/native-testing-and-evidence.md)).
5. **Installer** — `jdeskInstaller` runs `jpackage` on the app image to build the OS's
   native installer (DMG/PKG on macOS, MSI/EXE on Windows, DEB/RPM on Linux) into
   `build/jdesk/installer`. Type defaults to the OS default; override with
   `-PjdeskInstallerType=<dmg|pkg|msi|exe|deb|rpm>`. UNSIGNED unless a signing identity is
   configured. Verified end-to-end locally (real 34 MB DMG) and in the CI package jobs.

## Application name and the macOS menu bar

The name a user sees comes from `jdesk.applicationName`:

```kotlin
jdesk {
    applicationId.set("dev.example.dragon7")
    applicationName.set("Dragon 7")   // optional; defaults to the last applicationId segment
}
```

`jdeskPackage` passes it to `jpackage --name`, which becomes the app's **`CFBundleName`** — the
**bold application name** shown in the macOS menu bar (the item just right of the  menu). The task
also embeds it as a `-Djdesk.applicationName` launch option so the runtime's **`Quit <Name>`** menu
item matches the bold title exactly. Set it once and both agree; leave it unset and both derive from
the last `applicationId` segment (e.g. `dragon7`).

> **Only a packaged app can change the bold menu-bar name.** For a bundle-less dev launch
> (`./gradlew run`), macOS/AppKit forces that title to the executable name — it reads `java`, and no
> runtime call (`NSProcessInfo`, `setMainMenu:`, `-Xdock:name`) overrides it. The name is correct as
> soon as the app is packaged, because `CFBundleName` is then present. This affects only the cosmetic
> menu-bar title, never behaviour or identity (TCC/permissions read the code signature and bundle id).

## Build on the target OS only

Cross-packaging is **forbidden** ([spec 16.2](../../JDESK_CORE_FRAMEWORK_SPEC.md)): build
Windows packages on Windows, macOS on macOS, Linux on Linux. `jpackage` produces
platform-native images and cannot be trusted to cross-produce another OS's package. CI has
per-OS package jobs (`package-windows-x64`, `package-linux-x64`,
`package-macos-arm64`) and validates the installer payload structure.

Per-OS target formats (spec 16.2):

- **Windows:** application image + EXE or MSI.
- **macOS:** `.app` + DMG or PKG.
- **Linux:** application image + DEB (RPM may follow).

Both the **application image** and the **installer** steps are implemented: `jdeskPackage`
builds the image, `jdeskInstaller` builds the native installer on the same OS. macOS DMG is
verified locally; Windows MSI and Linux DEB are built in the CI package jobs.

## Signing (configuration hooks)

Signing is applied by the **OS toolchains**, wired through the plugin's signing hooks. The
spec ([16.3](../../JDESK_CORE_FRAMEWORK_SPEC.md)) requires a configuration surface and
documentation for:

- **Windows — Authenticode.** Sign the EXE/MSI with `signtool`, a code-signing certificate and an
  RFC 3161 timestamp; then require `signtool verify /pa` to pass.
- **macOS — Developer ID + notarization.** `codesign` with a Developer ID Application
  certificate, then `notarytool` submit + `stapler staple` + `stapler validate`.
- **Linux — artifact signing.** Emit an armored detached signature beside the DEB/RPM and require
  `gpg --verify` to pass. Repository metadata signing remains the distributor's responsibility.

**CI packages are `UNSIGNED`.** Unsigned packages are allowed for development verification
but are labeled `UNSIGNED` and **do not satisfy a signed-release gate**. A real signed
release requires the certificates/credentials above, which are not present in CI.

> **Honest status.** The plugin now wires and verifies every signing hook. The checked CI runners
> still have no Authenticode certificate or Apple Developer ID/notary profile, so no current CI
> artifact is publicly trusted or notarized. Do not treat an `UNSIGNED` artifact as signed.

## SBOM and checksums

Release-artifact hygiene lives in `jdesk-packager`'s `ReleaseArtifacts`
([spec 12.5, 16](../../JDESK_CORE_FRAMEWORK_SPEC.md)):

- **`writeChecksums(root, checksumFile)`** — SHA-256 for every file under `root`
  (recursively), emitted in GNU coreutils format, sorted for determinism.
- **`writeSbom(...)`** — a deterministic **CycloneDX 1.7** JSON SBOM listing the
  application, checksummed files and inspected runtime JARs with hashes, PURLs and a
  dependency graph. Composition is explicitly `incomplete`, rather than overstating
  knowledge of jlink/JDK internals.
  Deterministic (stable name-based serial number, no timestamps); unit-tested for validity
  and byte-stability.
- **`writeSpdxSbom(...)`** — the same inventory as an **SPDX 2.3** JSON document (application
  package, files, and library packages related by `DEPENDS_ON` / `CONTAINS`). Consumers and
  scanners differ on which format they ingest, and the EU Cyber Resilience Act makes a
  machine-readable SBOM binding from Sept 2026, so both are emitted. `SOURCE_DATE_EPOCH` (if
  set) keeps the SPDX `created` timestamp reproducible.

`jdeskPackage` invokes these after building the application image: it writes
`checksums.sha256` (over the produced image), `sbom.cyclonedx.json` and `sbom.spdx.json` next
to the image and logs `wrote checksums.sha256 (N files), sbom.cyclonedx.json and sbom.spdx.json
(UNSIGNED)`. Verified against a real 282-file jpackage app image. The redistributed `WebView2Loader.dll` license
is documented with Windows artifacts (see
[../platform/prerequisites.md](../platform/prerequisites.md)).

The release workflow already waits for the complete cross-platform `release-gate`, stages
library JARs and checksums, and has an opt-in OIDC provenance step controlled by
`JDESK_ENABLE_ATTESTATIONS`. Signed/notarized end-to-end distribution remains pending
until real credentials are supplied.
